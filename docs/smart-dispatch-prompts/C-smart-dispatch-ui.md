## 背景

演算法已經在 lib/matching/smartDispatch.ts 實作完成。現在要把它呈現給司機。

司機有兩種使用情境：
1. **未接單時**：瀏覽接單大廳，看到智慧排序後的推薦（按預期時薪）
2. **已接單時**：在「我的行程」頁面點某張卡片的「智慧排單」按鈕，跳到「配套推薦頁」

## 功能需求

不保留 tab 切換 UI，直接呈現智慧排序的結果。

---

### 2. 接單大廳「智慧排序」預設

#### 行為邏輯

當司機進入接單大廳：
- 如果司機有 1 張或多張已接單 → 進入「anchored 模式」，用每張已接單做配套推薦
- 如果司機無已接單 → 進入「standalone 模式」，按預期時薪排序所有訂單

#### API 設計

GET /api/driver/orders/smart-sort
回傳：
- mode: 'anchored' | 'standalone'
- anchors: Order[]（司機已接單列表）
- recommendations: Recommendation[]（綜合所有 anchor 的推薦）
- summary: { pairedCount, degradeBCount, degradeCCount }

當有多個 anchor 時，同一張候選訂單取對所有 anchor 中最高分那組作為顯示。

#### UI 結構

大廳分兩區：
- 上方「為你推薦的配套」：pairingScore === 100 或 warningFlag === 'DEGRADE_B' 的訂單
- 下方「其他可接訂單」：剩下的訂單（按時薪排序）

---

### 3. 推薦卡片設計

卡片要顯示：
- 類型徽章（🎯 配套推薦 / 💚 同區集中 / 🟠 較遠但配套）
- 總分（totalScore）
- 訂單基本資訊（時間、起終點、車型、價格）
- 推薦原因（matchReason）
- 銜接資訊（與哪張 anchor 配套、移動時間、時間緩衝）
- 接單按鈕

範例：
```
🎯 配套推薦 ｜ 分數 92
───────────────────
17:00 中和區 → 桃園機場
      MPV 5 人座 $700

💡 與你 14:00 板橋接機單配套
   移動時間 20 分鐘（相鄰區域）
   時間緩衝 2.5 小時

[ 接這單 → ]
```

| 降級 B | 💚 同區集中 | 藍色 | 次要色 |

### 6. 配套推薦專頁

路由：`/driver/smart-match?anchorId={orderId}`
