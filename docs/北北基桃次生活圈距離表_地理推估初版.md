## 一、設計原則

### 1.1 為什麼不納入宜蘭、新竹？

MVP 階段先聚焦於**北北基桃**，理由：

- **業務密度**：機場接送 90% 以上的單集中在北北基桃
- **資料品質**：行政區數量少、推估準確度較高
- **未來擴充**：宜蘭、新竹可作為第二階段加入

### 1.2 機場為什麼獨立成節點？

機場是接送業務的**核心錨點**，所有時間計算都圍繞它們。獨立成節點後：

- 計算更直觀（不用每次都判斷「大園區」=「桃機」）
- 未來如果要分 T1/T2，也容易擴充
- 智慧排單演算法可以直接用 `dropoffZone === 'TPE_AIRPORT'` 判斷

### 1.3 時間估算基準

- 採用**一般時段**（非尖峰、非離峰）的開車時間
- **尖峰加成**：早 7:30-9:30、晚 17:00-19:30 建議乘以 1.3
- **深夜減成**：23:00-翌 5:30 建議乘以 0.85
- 「同節點內部」一律給 15 分鐘（涵蓋區內移動 + 上下客時間）

// 完整距離矩陣（請依本文件第三~六節填入）
export const ZONE_DISTANCE_MATRIX: Record<ZoneCode, Record<ZoneCode, number>> = {
  // ... 完整內容請參考本文件 ...
};

/**
 * 取得兩節點之間的預估時間（分鐘）
 * @param from 起點節點代碼
 * @param to 終點節點代碼
 * @param peakMultiplier 尖峰加成（預設 1.0；尖峰建議 1.3）
 */
export function getZoneDistance(
  from: ZoneCode,
  to: ZoneCode,
  peakMultiplier: number = 1.0
): number {
  if (from === to) return 15;
  const base = ZONE_DISTANCE_MATRIX[from]?.[to] ?? 999;
  return Math.round(base * peakMultiplier);
}

/**
 * 判斷兩節點是否「鄰近」（用於智慧排單推薦）
 */
export function isNearby(from: ZoneCode, to: ZoneCode): boolean {
  return getZoneDistance(from, to) <= 30;
}
