# 派單中心重構規格書

## 日期
2026-04-23

## 背景

目前派單方的操作流程存在以下問題：
1. **AI 解析常常出錯**（時間、地點、車型、種類都解析錯）
2. **生成訂單時欄位不足**（LINE 訊息本來就只有時間 + 起訖點 + 金額，不足以構成完整訂單）
3. **上架流程不合理**：逼迫派單方在貼單時就要填完聯絡人、電話、人數、行李等詳細資訊，但 LINE 群組的訊息本來就沒有這些

LINE 群組真實作業流程是：車隊行控（派單方）貼出 brief 訊息，司機看到有興趣就接單，接單之後才跟行控聯繫取得詳細資訊。

新流程比照真實作業模式設計。

## 新完整流程

```
① 派單方貼 LINE 訊息（1600 送板橋 1200元）
② AI 解析核心欄位（時間、起訖點、金額、車型）
③ 派單方確認解析結果，上架
④ 司機看到訂單卡，按「申請接單」
⑤ 派單方收到通知，審查司機狀況
⑥ 派單方批准 + 填詳細資訊 → 司機正式接單
⑦（若拒絕）→ 訂單回到候選池，其他司機可繼續申請
⑧ 司機行程進入行程中心
```

### 與舊流程的差異

| 差異點 | 舊流程 | 新流程 |
|--------|--------|--------|
| 詳細資訊填寫時機 | 上架前（派單方強迫填完） | 上架後、批准司機時（派單方自行補填） |
| 司機接單方式 | 直接接單（先到先贏） | 申請接單（派單方審核） |
| 派單方把關能力 | 無，司機直接接走 | 有審核權，可拒絕不符合的司機 |
| 欄位完整性要求 | 上架前必須完整 | 上架前只需要核心欄位 |

## 司機申請接單的時機與 UI

- 司機在「首頁」看到所有已上架訂單的卡片牆
- 訂單卡顯示核心資訊（時間、起訖點、金額、車型）
- 司機按「申請接單」（不是「直接接單」）
- 按完後：訂單卡狀態變成「等待審核」，司機自己看不到这张订单了

## 派單方審核流程

### 收到通知
- 派單方 dashboard 出現紅色數字 badge，提示有新的申請
- 顯示在「訂單」或「待處理」 tab

### 審核面板（卡片擴展式，不換頁）
- 點進訂單卡 → 卡片展開，出現「司機審查面板」
- 面板內顯示：
  - **司機基本資訊**：姓名、電話、車牌、車型、顏色
  - **三證狀態**：駕照、行照、保險（是否已上傳）
  - **銀行帳號**：司機的收款帳戶
  - **歷史紀錄**：接單次數（系統有記錄的話）
- **操作按鍵**：批准 / 拒絕

### 批准時的動作
派單方按「批准」的同時，**必須填寫詳細資訊**，否則無法完成批准。

詳細資訊欄位與驗證規則：

| 欄位 | 必/選填 | 說明 |
|------|---------|------|
| 聯絡人姓名 | 必填 | — |
| 聯絡人電話 | 必填 | — |
| 航班編號 | 接機必填、送機選填 | 送機可不填 |
| 上車地址 | 送機必填、接機選填 | 送機時必填起點位址 |
| 下車地址 | 接機必填、送機選填 | 接機時必填終點位址 |
| 乘客人數 | 必填 | — |
| 行李件數 | 必填 | — |
| 特殊需求 | 選填 | 安全座椅、舉牌、行李上樓等 |
| 派單方備註 | 選填 | — |

派單方填完後按「確認批准」→ 訂單狀態變為「已接單」，司機行程中心可見。

### 拒絕時的動作
- 選取或填寫拒絕原因（選項：司機不符資格、訂單已派給別人、其他）
- 按「拒絕」→ 訂單回到候選池，其他司機可見

## AI 解析改革

### 簡化解析目標
AI 只解析**五個核心欄位**：
1. 時間（HH:MM）
2. 起點（接機 = 機場，送機 = 訊息中的地點）
3. 終點（接機 = 訊息中的地點，送機 = 機場）
4. 金額（數字）
5. 車型（小車/休旅/大車/任意）

**其餘所有資訊全部放 notes，不強迫解析。**

### 系統 Prompt 改革方向
- 車型統一用 `SEDAN_5` / `SUV_5` / `MPV_7` / `VAN_9` / `CUSTOM`
- 不再要求解析「聯絡人、電話、人數、行李」等欄位（這些是批准時才填的）
- 車牌（(L)(K)(R)(V)等）視為不影響解析的標記，放 notes 即可
- 若訊息格式完全無法解析 → rejected，讓派單方重新貼單

### 解析結果呈現
解析結果分三類：
- **已確認**：格式正確，五個核心欄位都有
- **需確認**：缺少 1-2 個核心欄位，派單方可在列表上直接修改
- **無法解析**：rejected，派單方需重新張貼

派單方在列表上可以**逐筆編輯核心欄位**（時間、起點、終點、金額、車型），確認後上架。

## 訂單狀態機

```
PENDING（草稿，AI 解析後）
  ↓
PUBLISHED（已上架，等待司機申請）
  ↓
ASSIGNED（已有司機申請，等待派單方審核）
  ↓（批准）
ACCEPTED（已接單，司機可見詳細資訊）
  ↓
IN_PROGRESS（行程開始）
  ↓
COMPLETED（行程完成）
```

若派單方拒絕：
```
ASSIGNED → PUBLISHED（回到候選池）
```

## 司機端的呈現

- 司機首頁只看到 `PUBLISHED` 狀態的訂單
- 按「申請接單」後，訂單從司機視角消失（避免重複申請）
- 申請成功後，該司機的 ASSIGNED 訂單出現在派單方的審核清單

## UI 設計方向

- 訂單卡：核心資訊為主（時間、起訖點、金額、車型）
- 司機審查面板：卡片向右或向下擴展，不換頁
- 派單方 dashboard：有「待處理」紅色 badge 提示
- 整體操作在一個頁面內完成，不需要跳轉

## 待確認的實務問題

1. **拒絕後的處理**：司機被拒絕後，會不會知道原因？還是其實不需要知道？
2. **一個訂單同時有多位司機申請**：怎麼處理？派單方只能批准一位，其他拒絕？
3. **時效性**：如果派單方一直不審核，訂單會不會過期？需要自動駁回或提醒？

## 實作優先順序

1. **Phase 1**：AI 解析改革（只解析五個核心欄位，系統 prompt 重寫）
2. **Phase 2**：派單方審核 flow（司機申請 → 派單方審查 → 批准並填寫詳細資訊）
3. **Phase 3**：司機端調整（申請接單按鍵、審核中狀態）
4. **Phase 4**：訂單狀態機完整實作

## 討論事項（已確認）

1. **拒絕通知**：司機被拒絕後，系統顯示「此行程派單方不同意指派」，不透露原因。

2. **多位司機申請機制**：
   - 司機按「申請接單」後，訂單立刻從接單大廳消失、其他司機看不見、也不在智慧排單候選名單
   - 派單方批准 → 給詳細資訊 → 訂單進入該司機的行程中心
   - 派單方拒絕 → 訂單回到接單大廳，智慧排單候選資格也恢復
   - 同時有多位司機申請的話：派單方只能批准一位，其他拒絕（其他人的申請自動失效）

3. **時效性**：
   - 若派單方不審核，訂單一直掛在等待審核狀態
   - 司機可透過訊息中心傳訊息給派單方詢問進度
   - 不需要自動提醒或自動駁回
