# 訂單解析 Skill

## 背景

台灣機場接送業界，車頭（派單方）會在 LINE 群組中發布訂單，格式極不統一。本 Skill 定義了解析規則，讓系統能從各種格式的原始文字中提取關鍵欄位，供司機一目瞭然地決定是否接單。

## 核心原則

解析訂單時，只需要精確提取 **五個關鍵欄位**，其餘內容全文保留到「備註」：

| 欄位 | 說明 | 司機決策用途 |
|------|------|-------------|
| 日期 | 服務日期 | 排班 |
| 時間 | 航班到達/出發時間，或上車時間 | 排班 |
| 種類 | 接機 / 送機 / 交通接駁 / 包車 | 決定要不要跑 |
| 車型 | 小車(5人) / 大車(7人) / 9座 / 任意 | 自己的車能不能跑 |
| 金額 | 新台幣金額 | 划不划算 |

**其餘所有資訊**（航班號、地址、人數、安全座椅、乘客姓名等）全部保留原文放入「備註」欄位，讓司機自己閱讀判斷。不需要逐欄位拆解這些次要資訊。

## 訂單種類定義

| 種類 | 關鍵字 | 說明 |
|------|--------|------|
| 接機 | 「接」+ 地點、航班號 + 「接」 | 從機場接人送到目的地 |
| 送機 | 「送」+ 機場簡稱（桃機/松機等） | 從出發地送人到機場 |
| 交通接駁 | 「交通接駁」、有「上」（上車點）「下」（下车點）但無航班 | 點對點接送，與機場無關 |
| 包車 | 「包車」、「時租」 | 按時間計費的包車服務 |

## 車型解析規則

車型是司機判斷「我的車能不能跑這單」的關鍵。解析邏輯：

| 原文關鍵字 | 解析為 | 說明 |
|-----------|--------|------|
| 小車、轎車、4人、5人 | 小車(5人座) | 一般轎車 |
| 休旅、7人、7座 | 休旅車(7人座) | 中型車 |
| 大車、9人、9座、8.9座 | 9人座 | 大型車。「8.9座」表示8座或9座都可以，統一歸為9人座 |
| 任意車 | 任意(不限車型) | R牌T牌都可以，什麼車型都行 |
| 任意R、任意(R) | 任意R牌 | 限租賃車，但不限車型大小 |
| 大車 | 9人座 | 注意：「大車」在業界通常指9人座 |

**重要的車型接單規則（寫進系統邏輯和 UI 提示）：**
- 大車可以接小車的單（9人座司機看得到5人座的訂單）
- 小車不能接大車的單（5人座司機看不到限定9人座的訂單）
- 「任意」和「任意車」代表所有車型都能接
- 「任意R」代表只有 R 牌車能接，但不限車型大小

## 車牌類型（R牌 vs T牌）

台灣營業車輛分兩種，平台必須分開標記：

| 類型 | 說明 |
|------|------|
| R牌 | 租賃車，車牌含 R 字母 |
| T牌 | 計程車，車牌含 T 字母 |

訂單中的標記方式：
- 「任意R」「(R)」→ 限 R 牌
- 「任意T」「(T)」→ 限 T 牌
- 「任意車」或未標記 → R 牌和 T 牌都可以

## 機場代碼與簡稱

| 正式代碼 | 全名 | 常見簡稱 |
|---------|------|---------|
| TPE | 桃園國際機場 | 桃機、桃園機場 |
| TSA | 臺北松山機場 | 松機、松山機場 |
| KHH | 高雄國際機場 | 小港 |
| RMQ | 臺中清泉崗機場 | 清泉崗、台中機場 |

## 特殊需求關鍵字

這些出現在原文中時，全部保留到備註欄位，不需要獨立解析成欄位：

| 關鍵字 | 含義 |
|--------|------|
| 反安、反安*N | 兒童安全座椅「反向安裝」，N 為數量。客人特別要求反方向安裝時才會標註 |
| 正安、正安*N | 兒童安全座椅「正向安裝」 |
| 安、安*N | 兒童安全座椅，未指定安裝方向 |
| 提籃 | 嬰兒提籃 |
| 舉牌 | 舉牌接機（舉名牌在入境大廳等候） |
| *N（地點後面） | 乘客人數 |

## 解析範例

以下是真實 LINE 訂單範例及其正確解析結果。開發解析功能時，用這些當測試案例。

### 範例 1：標準接機單

**原文：**
```
23:10 tr875 接北屯+北區 任意車2000
```

**解析結果：**
- time: "23:10"
- type: "pickup"
- vehicle: "any"
- plateType: "any"
- price: 2000
- notes: "航班 TR875 接北屯+北區"

### 範例 2：送機含特殊需求

**原文：**
```
3/28
06:00 楊梅*3送桃機 反安*2 9座 1700
```

**解析結果：**
- date: "3/28"
- time: "06:00"
- type: "dropoff"
- vehicle: "van9"
- plateType: "any"
- price: 1700
- notes: "楊梅*3送桃機 反安*2"

### 範例 3：含括號編號的批次訂單

**原文：**
```
(2)2310 接中正開封路一段 任意(R)($)800
```

**解析結果：**
- time: "23:10"
- type: "pickup"
- vehicle: "any_r"
- plateType: "R"
- price: 800
- notes: "#2 接中正開封路一段"

### 範例 4：精簡格式

**原文：**
```
3/25
小車
0500南港送 800
```

**解析結果：**
- date: "3/25"
- time: "05:00"
- type: "dropoff"
- vehicle: "small"
- plateType: "any"
- price: 800
- notes: "南港送機"

### 範例 5：交通接駁（非接送機）

**原文：**
```
3/28
21:00交通接駁
上-高雄巨蛋
下-文心崇德捷運站
+台中火車站
九座8人3200
```

**解析結果：**
- date: "3/28"
- time: "21:00"
- type: "transfer"
- vehicle: "van9"
- plateType: "any"
- price: 3200
- notes: "上:高雄巨蛋 下:文心崇德捷運站+台中火車站"

### 範例 6：大車接機

**原文：**
```
(one)(four)1800 接汐止長江街 大車 1000
```

**解析結果：**
- time: "18:00"
- type: "pickup"
- vehicle: "van9"
- plateType: "any"
- price: 1000
- notes: "#14 接汐止長江街"

## 格式變異與邊界情況

LINE 群組訂單沒有統一格式，以下是常見的變異：

1. **日期可能在訂單上方獨立一行**，也可能沒有日期（當天的單）
2. **時間格式不一**：23:10、2310、06:00、0500 都是合法時間
3. **金額位置不一**：通常在最後，但可能帶 $ 或 emoji 或什麼都不帶
4. **括號數字**：(one)(two) = 訂單編號，不影響解析
5. **多目的地**：用「+」連接，例如「北屯+北區」「西螺+斗六」
6. **車型可能在獨立一行**：例如先寫「小車」再列多筆訂單，代表以下都是小車
7. **一則訊息可能包含多筆訂單**：每行一筆，或用編號區分
8. **4位數字時間**：0500 表示 05:00，2310 表示 23:10
9. **航班號格式**：通常是2-3個英文字母 + 3-4個數字（tr875、it504、nx622）

## 解析優先順序

當原文模糊時，按此順序判斷：

1. 有航班號 → 接機或送機（再看有沒有「送」字）
2. 有「接」字 → 接機
3. 有「送」字 + 機場相關詞 → 送機
4. 有「交通接駁」或「上」「下」車點 → 交通接駁
5. 有「包車」「時租」 → 包車
6. 都沒有 → 標記為「pending」，放進備註讓車頭手動分類

## 系統實作對照表

### Prisma Order Model 欄位

| 欄位 | 說明 | 解析來源 |
|------|------|---------|
| type | pickup / dropoff / transfer / charter / pending | extractType() |
| vehicle | small / suv / van9 / any / any_r / pending | extractVehicle() |
| plateType | R / T / any | extractVehicle() |
| notes | 原文備註（含所有未解析的細節） | parseSingleLine() |
| rawText | 完整原始訊息 | 原始行文字 |
| scheduledTime | DateTime（日期+時間） | extractTime() + 預設日期 |

### 車型接單規則（系統邏輯）

| 司機車型 | 可接的訂單 vehicle | 不可接 |
|---------|-------------------|--------|
| 小車 (small/suv) | small, any | van9, any_r |
| 9人座 (van9) | small, suv, van9, any, any_r | - |
| R牌 (任意) | any, any_r | van9? 待確認 |

### 司機儀表板卡片顯示格式

```
抬頭列： 日期 + 種類 Badge + 金額
次行：   時間 + 航班號
車型行： 車型 Badge（顏色區分大小車）
備註區： 剩餘未解析資訊
```

## 開發指引

1. **先做 regex 解析**：用正則表達式處理時間、金額、車型等結構化欄位，先求能解析 80% 的訂單
2. **備註兜底**：任何解析不出來的內容，全部放備註，絕對不要丟掉任何原文資訊
3. **保留 rawText**：永遠存一份完整原文，方便日後改進解析邏輯時重新跑
4. **批次解析**：一則訊息可能有多筆訂單，要能拆開逐筆解析
5. **開發訂單相關功能前**，務必先讀本文件
