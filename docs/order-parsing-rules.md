# 訂單解析 Skill

## 背景

台灣機場接送業界，車頭（派單方）會在 LINE 群組中發布訂單，格式極不統一。本 Skill 定義了解析規則，讓系統能從各種格式的原始文字中提取關鍵欄位，供司機一目瞭然地決定是否接單。

## 核心原則

解析訂單時，只需要精確提取 **五個關鍵欄位**，其餘內容全文保留到「備註」：

| 欄位 | 說明 | 司機決策用途 |
|------|------|-------------|
| 日期 | 服務日期 | 排班 |
| 時間 | 航班到達/出發時間，或上車時間 | 排班 |
| 種類 | 接機 / 送機 / 交通接駁 / 包車 | 決定要不要跑 |
| 車型 | 小車(5人) / 大車(7人) / 9座 / 任意 | 自己的車能不能跑 |
| 金額 | 新台幣金額 | 划不划算 |

**其餘所有資訊**（航班號、地址、人數、安全座椅、乘客姓名等）全部保留原文放入「備註」欄位，讓司機自己閱讀判斷。不需要逐欄位拆解這些次要資訊。

## 訂單種類定義

| 種類 | 關鍵字 | 說明 |
|------|--------|------|
| 接機 | 「接」+ 地點、航班號 + 「接」 | 從機場接人送到目的地 |
| 送機 | 地點 +「送」（如「南港送」「板橋送」）、「送」+ 機場簡稱 | 從出發地送人到機場。注意：很多時候只寫「板橋送」而不會寫「板橋送桃機」，只要是「地點+送」的格式就是送機 |
| 交通接駁 | 「交通接駁」、有「上」（上車點）「下」（下车點）但無航班 | 點對點接送，與機場無關 |
| 包車 | 「包車」、「時租」 | 按時間計費的包車服務 |

## 車型解析規則

車型是司機判斷「我的車能不能跑這單」的關鍵。解析邏輯：

| 原文關鍵字 | 解析為 | 說明 |
|-----------|--------|------|
| 小車、轎車、4人、5人 | 小車(5人座) | 一般轎車 |
| 休旅、7人、7座 | 休旅車(7人座) | 中型車 |
| 大車、9人、9座、8.9座 | 9人座 | 大型車。「8.9座」表示8座或9座都可以，統一歸為9人座 |
| 任意車 | 任意(不限車型) | R牌T牌都可以，什麼車型都行 |
| 任意R、任意(R) | 任意R牌 | 限租賃車，但不限車型大小 |
| 大車 | 9人座 | 注意：「大車」在業界通常指9人座 |

**重要的車型接單規則（寫進系統邏輯和 UI 提示）：**
- 大車可以接小車的單（9人座司機看得到5人座的訂單）
- 小車不能接大車的單（5人座司機看不到限定9人座的訂單）
- 「任意」和「任意車」代表所有車型都能接
- 「任意R」代表只有 R 牌車能接，但不限車型大小

## 車牌類型（R牌 vs T牌）

台灣營業車輛分兩種，平台必須分開標記：

| 類型 | 說明 |
|------|------|
| R牌 | 租賃車，車牌含 R 字母 |
| T牌 | 計程車，車牌含 T 字母 |

訂單中的標記方式：
- 「任意R」「(R)」→ 限 R 牌
- 「任意T」「(T)」→ 限 T 牌
- 「任意車」或未標記 → R 牌和 T 牌都可以

## 機場代碼與簡稱

| 正式代碼 | 全名 | 常見簡稱 |
|---------|------|---------|
| TPE | 桃園國際機場 | 桃機、桃園機場 |
| TSA | 臺北松山機場 | 松機、松山機場 |
| KHH | 高雄國際機場 | 小港 |
| RMQ | 臺中清泉崗機場 | 清泉崗、台中機場 |

## 特殊需求關鍵字

這些出現在原文中時，全部保留到備註欄位，不需要獨立解析成欄位：

| 關鍵字 | 含義 |
|--------|------|
| 反安、反安*N | 兒童安全座椅「反向安裝」，N 為數量。客人特別要求反方向安裝時才會標註 |
| 正安、正安*N | 兒童安全座椅「正向安裝」 |
| 安、安*N | 兒童安全座椅，未指定安裝方向 |
| 提籃 | 嬰兒提籃 |
| 舉牌 | 舉牌接機（舉名牌在入境大廳等候） |
| *N（地點後面） | 乘客人數 |

## 格式變異與邊界情況

LINE 群組訂單沒有統一格式，以下是常見的變異：

1. **日期可能在訂單上方獨立一行**，也可能沒有日期（當天的單）
2. **時間格式不一**：23:10、2310、06:00、0500、0:10 都是合法時間
3. **金額位置不一**：通常在最後，但可能帶 $ 或 emoji 或什麼都不帶
4. **emoji 數字**：(one)(two) = 12，用圓括號包文字表示，當作訂單編號
5. **多目的地**：用「+」連接，例如「北屯+北區」「西螺+斗六」「永和送+永和」
6. **車型可能在獨立一行**：例如先寫「小車」再列多筆訂單，代表以下都是小車
7. **一則訊息可能包含多筆訂單**：每行一筆，或用編號區分
8. **4位數字時間**：0500 表示 05:00，2310 表示 23:10
9. **航班號格式**：通常是2-3個英文字母 + 3-4個數字（tr875、it504、nx622），但很多訂單沒有航班號

## 標題行繼承（最重要的解析模式）

這是業界最常見的貼單格式。車頭會在開頭用一行或數行設定「預設值」，後續每筆訂單繼承這些預設值，除非該行自己有標註覆蓋值。

**繼承規則：**
- 標題行設定的「日期」「車型」「車牌類型」「金額」為預設值
- 後續每一行是一筆獨立訂單
- 如果某行自己標註了金額（如 $800），則使用該行的金額，覆蓋預設金額
- 如果某行沒有標註金額，則使用標題行的預設金額
- 車型和車牌類型同理：有標註就覆蓋，沒標註就繼承

**完整範例：**

```
明天3/30 任意R $700
0:10接土城
0:35接大安+增高 $800
0:35接大安
0:55接淡水沙崙路 $800
1:00接萬華
1:30接板橋
3:15基隆送 $900
3:30瑞芳送 $1100
5:00文山送
5:30永和送+永和 $800
5:30信義送
6:00板橋送
6:30板橋送
7:00板橋送
7:30大安送
```

**解析邏輯：**

第一行「明天3/30 任意R $700」是標題行，設定預設值：
- 預設日期：3/30
- 預設車型：任意R牌
- 預設車牌類型：R
- 預設金額：700

後續每行解析結果：

| 原文 | 日期 | 時間 | 種類 | 車型 | 金額 | 說明 |
|------|------|------|------|------|------|------|
| 0:10接土城 | 3/30 | 00:10 | 接機 | 任意R牌 | 700 | 全部繼承預設 |
| 0:35接大安+增高 $800 | 3/30 | 00:35 | 接機 | 任意R牌 | **800** | 金額覆蓋預設 |
| 0:35接大安 | 3/30 | 00:35 | 接機 | 任意R牌 | 700 | 全部繼承預設 |
| 3:15基隆送 $900 | 3/30 | 03:15 | 送機 | 任意R牌 | **900** | 金額覆蓋預設 |
| 5:00文山送 | 3/30 | 05:00 | 送機 | 任意R牌 | 700 | 全部繼承預設 |
| 5:30永和送+永和 $800 | 3/30 | 05:30 | 送機 | 任意R牌 | **800** | 多目的地，金額覆蓋 |

**如何判斷哪一行是標題行：**
- 包含日期 + 車型 + 金額，但沒有「接」或「送」等動作詞 → 標題行
- 包含「明天」「後天」等相對日期詞 → 標題行
- 只有車型（如獨立一行「小車」）→ 也是標題行（設定車型預設值）
- 有時間 + 有「接」或「送」→ 訂單行，不是標題行

## 解析優先順序

當原文模糊時，按此順序判斷：

1. 有航班號 → 接機或送機（再看有沒有「送」字）
2. 有「接」字 + 地點 → 接機（如「接土城」「接萬華」）
3. 有地點 +「送」字 → 送機（如「板橋送」「南港送」）。注意：不一定會寫目標機場名稱
4. 有「送」字 + 機場簡稱 → 送機（如「送桃機」「送松機」）
5. 有「交通接駁」或「上」「下」車點 → 交通接駁
6. 有「包車」「時租」 → 包車
7. 都沒有 → 標記為「pending」，放進備註讓車頭手動分類

## 資料庫欄位建議

開發 Prisma schema 時，建議用「批次」概念管理：車頭每次貼單是一個批次，批次記錄預設值，訂單記錄每筆細節。

```prisma
// 車頭每次貼單建立一個批次
model OrderBatch {
  id            String   @id @default(cuid())
  dispatcherId  String   // 車頭ID
  date          DateTime // UI 選的日期
  type          String   // pickup / dropoff / transfer / charter（UI 選的種類）
  vehicle       String   // small / suv / van9 / any / any_r（UI 選的車型）
  plateType     String   // R / T / any（UI 選的車牌）
  defaultPrice  Int?     // 預設金額（選填，null 表示每筆自帶）
  rawText       String   // 完整貼入的原始文字（保留原文）
  createdAt     DateTime @default(now())
  orders        Order[]
}

// 每筆解析出的訂單
model Order {
  id          String     @id @default(cuid())
  batchId     String     // 所屬批次
  batch       OrderBatch @relation(fields: [batchId], references: [id])
  date        DateTime   // 繼承自批次
  time        String     // 從文字解析出的時間
  type        String     // 繼承自批次
  pickup      String     // 上車點（接機 = 機場，送機 = 從文字提取的地點）
  dropoff     String     // 下車點（接機 = 從文字提取的地點，送機 = 機場）
  vehicle     String     // 繼承自批次
  plateType   String     // 繼承自批次
  price       Int        // 行內金額 ?? 批次預設金額
  priceSource String     // "default"（繼承）或 "inline"（行內覆蓋）或 "edited"（車頭手動修改）
  notes       String     // 備註（扣除時間、地點、金額後的剩餘文字）
  rawLine     String     // 該行的原始文字
  status      String     @default("pending") // pending / assigned / in_progress / completed / cancelled
  driverId    String?    // 接單司機ID
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt
}
```

## 車頭貼單介面設計

### 完整流程（三步驟）

```
步驟一：設定預設值 → 步驟二：貼單+解析 → 步驟三：確認+編輯+上架
```

車頭貼單分三步驟。必選欄位沒選完，後續步驟鎖定。確認後立刻上牆，司機馬上看到。

---

### 步驟一：設定預設值

```
┌─────────────────────────────────────────────┐
│  日期：[  3/30（日）      ▼ ]  ← 必選，日曆選擇  │
│  種類：[ 接機 | 送機 | 交通接駁 | 包車 ]  ← 必選  │
│  車型：[ 小車 | 休旅 | 9座 | 任意 | 任意R ] ← 必選 │
│  車牌：[ R牌 | T牌 | 不限 ]  ← 必選             │
│  預設金額：[ 700 ] （選填，空白表示每筆自帶金額）   │
└─────────────────────────────────────────────┘
```

**預設值鎖定規則：**

| 欄位 | 是否必選 | 鎖定行為 |
|------|---------|---------|
| 日期 | 必選 | 未選擇 → 文字框顯示「請先選擇日期」，不可輸入 |
| 種類 | 必選 | 未選擇 → 文字框顯示「請先選擇種類」，不可輸入 |
| 車型 | 必選 | 未選擇 → 文字框顯示「請先選擇車型」，不可輸入 |
| 車牌 | 必選 | 跟隨車型選擇自動帶入（選「任意R」→ 自動選 R牌） |
| 預設金額 | 選填 | 空白 = 每筆訂單必須自帶金額，解析時沒金額的行標記警告 |

**種類整批統一：** 一次貼單只能選一種種類。如果車頭同時有接機和送機的單，需要分兩批貼。

---

### 步驟二：貼單 + 解析

預設值選完後解鎖文字框，車頭從 LINE 複製訂單整批貼入，按「解析預覽」。

```
┌─────────────────────────────────────────┐
│ 0:10接土城                               │
│ 0:35接大安+增高 $800                      │
│ 0:35接大安                               │
│ 0:55接淡水沙崙路 $800                     │
│ 1:00接萬華                               │
│ 1:30接板橋                               │
└─────────────────────────────────────────┘
              [ 解析預覽 ]
```

---

### 步驟三：確認 + 編輯 + 上架（核心確認機制）

解析完成後進入確認頁面。這是最關鍵的步驟。

**頁面頂部：** 顯示預設值標籤（讓車頭再次確認預設值沒選錯）

```
預設值：[3/30] [接機] [任意R] [$700]
```

**確認清單功能：**

1. **逐筆勾選**：每筆訂單前面有勾選框（checkbox），預設全部勾選。車頭可以取消不想上架的訂單。
2. **全選/取消全選**：頂部有全選開關，方便操作。
3. **逐筆編輯**：每筆訂單的「金額」和「備註」可以直接點擊修改。
4. **警告標記**：有問題的訂單標 ⚠️，車頭必須處理。
5. **上架計數**：底部即時顯示「已選 N 筆 / 共 M 筆」。
6. **上架按鈕**：按下後，只有勾選的訂單會上牆。

```
┌─────────────────────────────────────────────┐
│ ☐ 全選                          確認訂單     │
│                                              │
│ ┌──────────────────────────────────────────┐ │
│ │ ☑ 00:10  接機                            │ │
│ │   桃園機場 → [土城        ]               │ │
│ │   備註：[             ]    金額：[$700  ] │ │
│ └──────────────────────────────────────────┘ │
│                                              │
│ ┌──────────────────────────────────────────┐ │
│ │ ☑ 00:35  接機                 (行內金額)  │ │
│ │   桃園機場 → [大安+增高   ]               │ │
│ │   備註：[             ]    金額：[$800  ] │ │
│ └──────────────────────────────────────────┘ │
│                                              │
│ ┌──────────────────────────────────────────┐ │
│ │ ☐ 06:00  送機          ← 取消勾選 = 不上架 │ │
│ │   [板橋          ] → 桃園機場             │ │
│ │   備註：[             ]    金額：[$700  ] │ │
│ └──────────────────────────────────────────┘ │
│                                              │
│ 已選 2 筆 / 共 3 筆    [返回修改] [上架到牆上] │
└─────────────────────────────────────────────┘
```

**確認頁面的行為規則：**

| 操作 | 行為 |
|------|------|
| 取消勾選 | 該筆訂單變灰、虛線邊框，不會上架但也不會刪除原文 |
| 修改金額 | 直接在輸入框改數字，priceSource 自動變為 "edited" |
| 修改備註 | 直接在輸入框改文字 |
| 修改地點 | 接機可改目的地，送機可改出發地。機場那端固定不可改 |
| 返回修改 | 回到步驟二的文字框，可以重新貼單或修改文字 |
| 上架到牆上 | 只有勾選的訂單寫入資料庫，status = "pending"，司機立刻看到 |

**警告標記規則：**

| 情況 | 顯示 | 是否阻擋上架 |
|------|------|-------------|
| 沒有預設金額，行內也沒金額 | ⚠️「未設定金額」，金額欄位紅色邊框 | 阻擋，必須手動填入金額 |
| 無法解析出時間 | ⚠️「無法辨識時間」 | 阻擋，必須手動修正 |
| 無法提取地點 | ⚠️「無法辨識地點，請手動填寫」，地點欄位紅色邊框 | 阻擋，必須手動填入 |
| 行內金額與預設金額差異超過 50% | ⚠️「金額差異較大，請確認」 | 不阻擋，僅提醒 |
| 空行或純文字行（無時間） | 自動跳過，不顯示在清單中 | 不顯示 |

**上架成功後：**
- 顯示成功訊息：「N 筆訂單已上架，司機現在可以看到並接單」
- 提供兩個按鈕：「繼續貼單」（回到步驟一）和「查看已上架訂單」（跳到訂單管理頁）
- 上架的訂單 status 設為 "pending"，等待司機接單

## 簡化後的解析邏輯

因為車頭已經手動選好「日期」「種類」「車型」「車牌」「預設金額」，解析器需要從每一行文字中提取：

1. **時間**（必須有，沒有時間的行標記錯誤）
2. **地點**（從「接XX」或「XX送」中提取上下車點）
3. **金額覆蓋**（如果行內有金額，覆蓋預設金額）
4. **備註**（整行扣除時間、地點、金額後的剩餘文字）

### 解析器做四件事

```
輸入：一行文字 + 預設值（來自 UI 選擇）
輸出：
{
  date:      預設值.日期,
  time:      從文字提取,
  type:      預設值.種類,
  pickup:    根據種類決定（見下方邏輯）,
  dropoff:   根據種類決定（見下方邏輯）,
  vehicle:   預設值.車型,
  plateType: 預設值.車牌,
  price:     行內金額 ?? 預設值.金額,
  notes:     剩餘文字,
  rawText:   原始整行文字
}
```

### 地點提取邏輯

地點提取根據「種類」（車頭在步驟一已選好）決定上下車點：

**接機（pickup = 機場，dropoff = 從文字提取）：**
```
原文：「0:10接土城」
→ 去掉時間 → 「接土城」
→ 去掉「接」字 → 地點 = 「土城」
→ pickup = "桃園機場"（預設，根據平台設定）
→ dropoff = "土城"
```

**送機（pickup = 從文字提取，dropoff = 機場）：**
```
原文：「6:00板橋送」
→ 去掉時間 → 「板橋送」
→ 去掉「送」字 → 地點 = 「板橋」
→ pickup = "板橋"
→ dropoff = "桃園機場"（預設）
```

**多目的地（用「+」連接）：**
```
原文：「0:35接大安+增高」
→ dropoff = "大安+增高"（整段保留，不拆開）

原文：「5:30永和送+永和」
→ pickup = "永和+永和"（整段保留）
```

**地點提取的 regex 規則：**
- 接機：去掉開頭時間後，匹配「接」後面到金額或行尾的文字 → 那就是 dropoff
- 送機：去掉開頭時間後，匹配「送」前面的文字（不含時間）→ 那就是 pickup
- 如果提取不到地點 → pickup/dropoff 設為 null，整行文字放備註，車頭在確認頁面手動補填

### 每行文字的完整解析步驟

```
原文：「0:35接大安+增高 $800」
種類預設值：接機

Step 1 → 提取時間：找開頭的數字時間格式 → 00:35
Step 2 → 提取金額：找 $數字 或 行尾數字 → 800（覆蓋預設）
Step 3 → 提取地點：去掉時間和金額 → 「接大安+增高」→ 去掉「接」→ dropoff = 「大安+增高」，pickup = 「桃園機場」
Step 4 → 剩餘文字當備註 → （本例無剩餘）
```

```
原文：「6:00板橋送」
種類預設值：送機

Step 1 → 提取時間 → 06:00
Step 2 → 提取金額 → 沒有 → 用預設金額
Step 3 → 提取地點 → 「板橋送」→ 去掉「送」→ pickup = 「板橋」，dropoff = 「桃園機場」
Step 4 → 剩餘文字當備註 → （本例無剩餘）
```

```
原文：「1:00接萬華」
種類預設值：接機

Step 1 → 提取時間 → 01:00
Step 2 → 提取金額 → 沒有 → 用預設金額
Step 3 → 提取地點 → 「接萬華」→ 去掉「接」→ dropoff = 「萬華」，pickup = 「桃園機場」
Step 4 → 剩餘文字當備註 → （本例無剩餘）
```

## UI 顯示建議（司機端訂單卡片）

司機看卡片的決策順序：種類 → 上下車點（跟現有行程有沒有順路）→ 金額 → 車型 → 時間。卡片設計要對應這個順序：

**接機卡片：**
```
┌─────────────────────────────────┐
│ 接機         3/30（日） 00:10   │  ← 種類 + 日期時間
│                                 │
│ 桃園機場 → 土城                  │  ← 上下車點（最醒目）
│                                 │
│ $700          任意R牌            │  ← 金額 + 車型
│ ─────────────────────────────── │
│ 備註：（無）                     │  ← 其他資訊
└─────────────────────────────────┘
```

**送機卡片：**
```
┌─────────────────────────────────┐
│ 送機         3/30（日） 06:00   │
│                                 │
│ 板橋 → 桃園機場                  │  ← 出發地 → 機場
│                                 │
│ $700          任意R牌            │
│ ─────────────────────────────── │
│ 備註：（無）                     │
└─────────────────────────────────┘
```

**多目的地接機卡片：**
```
┌─────────────────────────────────┐
│ 接機         3/30（日） 00:35   │
│                                 │
│ 桃園機場 → 大安+增高             │  ← 多點用 + 連接
│                                 │
│ $800          任意R牌            │
│ ─────────────────────────────── │
│ 備註：（無）                     │
└─────────────────────────────────┘
```

**交通接駁卡片：**
```
┌─────────────────────────────────┐
│ 交通接駁      3/28（六） 21:00  │
│                                 │
│ 高雄巨蛋 → 文心崇德捷運站+台中火車站│
│                                 │
│ $3,200        9人座             │
│ ─────────────────────────────── │
│ 備註：8人                       │
└─────────────────────────────────┘
```

**卡片設計原則：**
- 上下車點用箭頭「→」連接，是卡片視覺最大最醒目的一行
- 接機：上車點固定顯示機場名稱（根據平台預設或訂單指定）
- 送機：下车點固定顯示機場名稱
- 如果地點提取失敗（pickup/dropoff 為 null），顯示備註原文讓司機自行判斷

## 開發指引

1. **UI 先行**：先做車頭的預設值選擇介面（日期、種類、車型、車牌、預設金額），三個必選沒填完就鎖定文字框
2. **解析器極簡化**：因為預設值已由 UI 提供，解析器只需從每行提取「時間」和「可選的金額覆蓋」，剩下全當備註
3. **確認機制必做**：解析完必須進確認頁面，支援逐筆勾選、金額編輯、備註編輯，車頭確認後才上架
4. **備註兜底**：扣除時間和金額後的文字全部放備註，絕對不要丟掉任何原文資訊
5. **保留 rawText**：永遠存一份完整原文，方便日後改進解析邏輯時重新跑
6. **批次解析**：文字框的內容按換行拆開，每行一筆訂單，空行跳過
7. **上架即時生效**：確認上架後，訂單 status = "pending"，司機端立刻看到
8. **之後可接 AI**：初期用 regex 提取時間和金額就夠了，穩定後可以加 LLM 輔助解析備註中的地址和特殊需求

## API 路由設計

整個貼單流程需要兩個 API：

### POST /api/orders/parse（解析，不寫入資料庫）

前端步驟二按「解析預覽」時呼叫。只做解析，回傳預覽資料讓車頭確認。

```
Request:
{
  defaults: { date, type, vehicle, plateType, defaultPrice },
  rawText: "0:10接土城\n0:35接大安+增高 $800\n..."
}

Response:
{
  success: true,
  data: {
    orders: [
      { time: "00:10", pickup: "桃園機場", dropoff: "土城", price: 700, priceSource: "default", notes: "", rawLine: "0:10接土城", warnings: [] },
      { time: "00:35", pickup: "桃園機場", dropoff: "大安+增高", price: 800, priceSource: "inline", notes: "", rawLine: "0:35接大安+增高 $800", warnings: [] },
      { time: null, pickup: null, dropoff: null, price: null, notes: "...", rawLine: "...", warnings: ["無法辨識時間"] }
    ],
    totalCount: 3,
    warningCount: 1
  }
}
```

### POST /api/orders/publish（上架，寫入資料庫）

前端步驟三按「上架到牆上」時呼叫。只寫入車頭勾選的訂單。

```
Request:
{
  batch: { date, type, vehicle, plateType, defaultPrice, rawText },
  orders: [
    { time: "00:10", pickup: "桃園機場", dropoff: "土城", price: 700, priceSource: "default", notes: "", rawLine: "0:10接土城" },
    { time: "00:35", pickup: "桃園機場", dropoff: "大安+增高", price: 800, priceSource: "edited", notes: "備註被改過", rawLine: "0:35接大安+增高 $800" }
  ]
}

Response:
{
  success: true,
  data: {
    batchId: "clxx...",
    publishedCount: 2,
    message: "2 筆訂單已上架"
  }
}
```

注意：前端只發送車頭勾選的訂單，未勾選的不包含在 request 中。
