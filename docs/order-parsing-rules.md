# 訂單解析 Skill

## 背景

台灣機場接送業界，車頭（派單方）會在 LINE 群組中發布訂單，格式極不統一。本 Skill 定義了解析規則，讓系統能從各種格式的原始文字中提取關鍵欄位，供司機一目瞭然地決定是否接單。

## 核心原則

解析訂單時，只需要精確提取 **五個關鍵欄位**，其餘內容全文保留到「備註」：

| 欄位 | 說明 | 司機決策用途 |
|------|------|-------------|
| 日期 | 服務日期 | 排班 |
| 時間 | 航班到達/出發時間，或上車時間 | 排班 |
| 種類 | 接機 / 送機 / 交通接駁 / 包車 | 決定要不要跑 |
| 車型 | 小車(5人) / 大車(7人) / 9座 / 任意 | 自己的車能不能跑 |
| 金額 | 新台幣金額 | 划不划算 |

**其餘所有資訊**（航班號、地址、人數、安全座椅、乘客姓名等）全部保留原文放入「備註」欄位，讓司機自己閱讀判斷。不需要逐欄位拆解這些次要資訊。

## 訂單種類定義

| 種類 | 關鍵字 | 說明 |
|------|--------|------|
| 接機 | 「接」+ 地點、航班號 + 「接」 | 從機場接人送到目的地 |
| 送機 | 地點 +「送」（如「南港送」「板橋送」）、「送」+ 機場簡稱 | 從出發地送人到機場。注意：很多時候只寫「板橋送」而不會寫「板橋送桃機」，只要是「地點+送」的格式就是送機 |
| 交通接駁 | 「交通接駁」、有「上」（上車點）「下」（下車點）但無航班 | 點對點接送，與機場無關 |
| 包車 | 「包車」、「時租」 | 按時間計費的包車服務 |

## 車型解析規則

車型是司機判斷「我的車能不能跑這單」的關鍵。解析邏輯：

| 原文關鍵字 | 解析為 | 說明 |
|-----------|--------|------|
| 小車、轎車、4人、5人 | 小車(5人座) | 一般轎車 |
| 休旅、7人、7座 | 休旅車(7人座) | 中型車 |
| 大車、9人、9座、8.9座 | 9人座 | 大型車。「8.9座」表示8座或9座都可以，統一歸為9人座 |
| 任意車 | 任意(不限車型) | R牌T牌都可以，什麼車型都行 |
| 任意R、任意(R) | 任意R牌 | 限租賃車，但不限車型大小 |
| 大車 | 9人座 | 注意：「大車」在業界通常指9人座 |

**重要的車型接單規則（寫進系統邏輯和 UI 提示）：**
- 大車可以接小車的單（9人座司機看得到5人座的訂單）
- 小車不能接大車的單（5人座司機看不到限定9人座的訂單）
- 「任意」和「任意車」代表所有車型都能接
- 「任意R」代表只有 R 牌車能接，但不限車型大小

## 車牌類型（R牌 vs T牌）

台灣營業車輛分兩種，平台必須分開標記：

| 類型 | 說明 |
|------|------|
| R牌 | 租賃車，車牌含 R 字母 |
| T牌 | 計程車，車牌含 T 字母 |

訂單中的標記方式：
- 「任意R」「(R)」→ 限 R 牌
- 「任意T」「(T)」→ 限 T 牌
- 「任意車」或未標記 → R 牌和 T 牌都可以

## 機場代碼與簡稱

| 正式代碼 | 全名 | 常見簡稱 |
|---------|------|---------|
| TPE | 桃園國際機場 | 桃機、桃園機場 |
| TSA | 臺北松山機場 | 松機、松山機場 |
| KHH | 高雄國際機場 | 小港 |
| RMQ | 臺中清泉崗機場 | 清泉崗、台中機場 |

## 特殊需求關鍵字

這些出現在原文中時，全部保留到備註欄位，不需要獨立解析成欄位：

| 關鍵字 | 含義 |
|--------|------|
| 反安、反安*N | 兒童安全座椅「反向安裝」，N 為數量。客人特別要求反方向安裝時才會標註 |
| 正安、正安*N | 兒童安全座椅「正向安裝」 |
| 安、安*N | 兒童安全座椅，未指定安裝方向 |
| 提籃 | 嬰兒提籃 |
| 舉牌 | 舉牌接機（舉名牌在入境大廳等候） |
| *N（地點後面） | 乘客人數 |

## 格式變異與邊界情況

LINE 群組訂單沒有統一格式，以下是常見的變異：

1. **日期可能在訂單上方獨立一行**，也可能沒有日期（當天的單）
2. **時間格式不一**：23:10、2310、06:00、0500、0:10 都是合法時間
3. **金額位置不一**：通常在最後，但可能帶 $ 或 💲 或什麼都不帶
4. **emoji 數字**：(one)(two) = 12，用圓括號包文字表示，當作訂單編號
5. **多目的地**：用「+」連接，例如「北屯+北區」「西螺+斗六」「永和送+永和」
6. **車型可能在獨立一行**：例如先寫「小車」再列多筆訂單，代表以下都是小車
7. **一則訊息可能包含多筆訂單**：每行一筆，或用編號區分
8. **4位數字時間**：0500 表示 05:00，2310 表示 23:10
9. **航班號格式**：通常是2-3個英文字母 + 3-4個數字（tr875、it504、nx622），但很多訂單沒有航班號

## 標題行繼承（最重要的解析模式）

這是業界最常見的貼單格式。車頭會在開頭用一行或數行設定「預設值」，後續每筆訂單繼承這些預設值，除非該行自己有標註覆蓋值。

**繼承規則：**
- 標題行設定的「日期」「車型」「車牌類型」「金額」為預設值
- 後續每一行是一筆獨立訂單
- 如果某行自己標註了金額（如 $800），則使用該行的金額，覆蓋預設金額
- 如果某行沒有標註金額，則使用標題行的預設金額
- 車型和車牌類型同理：有標註就覆蓋，沒標註就繼承

**完整範例：**

```
明天3/30 任意R $700
0:10接土城
0:35接大安+增高 $800
0:35接大安
0:55接淡水沙崙路 $800
1:00接萬華
1:30接板橋
3:15基隆送 $900
3:30瑞芳送 $1100
5:00文山送
5:30永和送+永和 $800
5:30信義送
6:00板橋送
6:30板橋送
7:00板橋送
7:30大安送
```

**解析邏輯：**

第一行「明天3/30 任意R $700」是標題行，設定預設值：
- 預設日期：3/30
- 預設車型：任意R牌
- 預設車牌類型：R
- 預設金額：700

後續每行解析結果：

| 原文 | 日期 | 時間 | 種類 | 車型 | 金額 | 說明 |
|------|------|------|------|------|------|------|
| 0:10接土城 | 3/30 | 00:10 | 接機 | 任意R牌 | 700 | 全部繼承預設 |
| 0:35接大安+增高 $800 | 3/30 | 00:35 | 接機 | 任意R牌 | **800** | 金額覆蓋預設 |
| 0:35接大安 | 3/30 | 00:35 | 接機 | 任意R牌 | 700 | 全部繼承預設 |
| 3:15基隆送 $900 | 3/30 | 03:15 | 送機 | 任意R牌 | **900** | 金額覆蓋預設 |
| 5:00文山送 | 3/30 | 05:00 | 送機 | 任意R牌 | 700 | 全部繼承預設 |
| 5:30永和送+永和 $800 | 3/30 | 05:30 | 送機 | 任意R牌 | **800** | 多目的地，金額覆蓋 |

**如何判斷哪一行是標題行：**
- 包含日期 + 車型 + 金額，但沒有「接」或「送」等動作詞 → 標題行
- 包含「明天」「後天」等相對日期詞 → 標題行
- 只有車型（如獨立一行「小車」）→ 也是標題行（設定車型預設值）
- 有時間 + 有「接」或「送」→ 訂單行，不是標題行

## 解析優先順序

當原文模糊時，按此順序判斷：

1. 有航班號 → 接機或送機（再看有沒有「送」字）
2. 有「接」字 + 地點 → 接機（如「接土城」「接萬華」）
3. 有地點 +「送」字 → 送機（如「板橋送」「南港送」）。注意：不一定會寫目標機場名稱
4. 有「送」字 + 機場簡稱 → 送機（如「送桃機」「送松機」）
5. 有「交通接駁」或「上」「下」車點 → 交通接駁
6. 有「包車」「時租」 → 包車
7. 都沒有 → 標記為「待確認」，放進備註讓車頭手動分類

## 資料庫欄位建議

開發 Prisma schema 時，建議用「批次」概念管理：車頭每次貼單是一個批次，批次記錄預設值，訂單記錄每筆細節。

```prisma
// 車頭每次貼單建立一個批次
model OrderBatch {
  id            String   @id @default(cuid())
  dispatcherId  String   // 車頭ID
  date          DateTime // UI 選的日期（預設值）
  defaultPrice  Int?     // 預設金額（選填）
  kenichiRequired Boolean @default(false) // 是否需要肯驛系統
  rawText       String   // 完整貼入的原始文字（保留原文）
  createdAt     DateTime @default(now())
  orders        Order[]
}

// 每筆解析出的訂單
model Order {
  id          String     @id @default(cuid())
  batchId     String     // 所屬批次
  batch       OrderBatch @relation(fields: [batchId], references: [id])
  date        DateTime   // 繼承自批次（文字中有日期則覆蓋）
  time        String     // 從文字解析
  type        String     // pickup/dropoff/transfer/charter/package — 自動判斷
  airport     String?    // 機場代碼 TPE/TSA/KHH/RMQ — 自動辨識
  pickup      String     // 上車點 — 自動判斷
  dropoff     String     // 下車點 — 自動判斷
  vehicle     String?    // small/suv/van9/any/any_r — 自動從標題行偵測
  plateType   String?    // R/T/any — 自動從標題行偵測
  price       Int        // 行內金額 ?? 批次預設金額
  priceSource String     // "default" / "inline" / "edited"
  kenichiRequired Boolean @default(false) // 繼承自批次
  notes       String     // 備註（所有非必備資訊的原文）
  rawLine     String     // 該行的原始文字
  status      String     @default("pending") // pending / assigned / in_progress / completed / cancelled
  transferStatus String  @default("pending") // pending / completed（轉帳狀態）
  driverId    String?    // 接單司機ID
  createdAt   DateTime   @default(now())
  updatedAt   DateTime   @updatedAt
}
```

## 車頭貼單介面設計

### 完整流程（三步驟）

```
步驟一：設定預設值 → 步驟二：貼單+解析 → 步驟三：確認+編輯+上架
```

車頭貼單分三步驟。必選欄位沒選完，後續步驟鎖定。確認後立刻上牆，司機馬上看到。

---

### 步驟一：設定預設值

大部分欄位由系統從文字自動判斷，車頭只需設定少量預設值：

```
┌─────────────────────────────────────────────┐
│  日期：[  4/1（二）       ▼ ]  ← 必選，日曆選擇 │
│  預設金額：[ 700 ] （選填，空白表示每筆自帶金額）  │
│  ☐ 肯驛系統（勾選後所有訂單標記為肯驛單）         │
└─────────────────────────────────────────────┘
```

**以下欄位全部由系統從訂單文字自動判斷，不需要車頭手動選：**
- 種類（接機/送機/交通接駁/包車）
- 機場（桃機/松山/小港/清泉崗）
- 車型（任意/休旅/大車/小車/9座等）
- 車牌限制（任意R/限V 等）
- 上下車點

**預設值規則：**

| 欄位 | 是否必選 | 說明 |
|------|---------|------|
| 日期 | 必選 | 未選擇 → 文字框鎖定。文字中如果有日期會覆蓋此預設值 |
| 預設金額 | 選填 | 空白 = 每筆訂單必須自帶金額，沒金額的行標記錯誤 |
| 肯驛系統 | 選填 | 勾選 → 此批訂單標記為肯驛單 |

---

### 步驟二：貼單 + 解析

預設值選完後解鎖文字框，車頭從 LINE 複製訂單整批貼入，按「解析預覽」。

```
┌─────────────────────────────────────────┐
│ 0:10接土城                               │
│ 0:35接大安+增高 $800                      │
│ 0:35接大安                               │
│ 0:55接淡水沙崙路 $800                     │
│ 1:00接萬華                               │
│ 1:30接板橋                               │
└─────────────────────────────────────────┘
              [ 解析預覽 ]
```

---

### 步驟三：確認 + 編輯 + 上架（核心確認機制）

解析完成後進入確認頁面。訂單分成兩區：**解析成功** 和 **解析失敗（需補正）**。

**頁面頂部：** 顯示解析摘要

```
預設值：[4/1] [$700]
解析結果：共 10 筆，成功 8 筆，需補正 2 筆
```

**六個必備欄位，缺一不可：**

| 欄位 | 缺少時的處理 |
|------|-------------|
| 日期 | 繼承 UI 預設日期，不會缺 |
| 時間 | 解析失敗，進入「需補正」區 |
| 種類 | 解析失敗，進入「需補正」區 |
| 起點 | 解析失敗，進入「需補正」區 |
| 終點 | 解析失敗，進入「需補正」區 |
| 金額 | 無預設金額且行內也沒有 → 進入「需補正」區 |

---

```
│                                              │
│ ┌──────────────────────────────────────────┐ │
│ │ ✗ [    ▼]  [    ▼]    缺少：時間、種類    │ │
│ │   原文：「某段無法解析的文字」               │ │
│ │   ...（車頭手動補填所有欄位）                │ │
│ │                                            │ │
│ │   [補正完成，移到成功區]  [放棄此筆]          │ │
│ └──────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
```

**確認頁面的行為規則：**

| 操作 | 行為 |
|------|------|
| 取消勾選（成功區）| 該筆訂單變灰、虛線邊框，不會上架 |
| 修改種類 | 下拉選單（接機/送機/交通接駁/包車），系統自動判斷但車頭可以改 |
| 修改車型 | 下拉選單（任意/小車/休旅/大車/9座等），系統自動判斷但車頭可以改 |
| 修改金額 | 直接在輸入框改數字 |
| 修改備註 | 直接在輸入框改文字 |
| 修改起點/終點 | 直接在輸入框改文字 |
| 補正完成（失敗區）| 所有必備欄位填完後，按鈕把該筆移到成功區 |
| 放棄此筆（失敗區）| 不要這筆訂單，從清單移除 |
| 返回修改 | 回到步驟二的文字框 |
| 上架到牆上 | 只有成功區中勾選的訂單寫入資料庫，status = "pending"，司機立刻看到 |

**上架成功後：**
- 顯示成功訊息：「N 筆訂單已上架，司機現在可以看到並接單」
- 提供兩個按鈕：「繼續貼單」（回到步驟一）和「查看已上架訂單」（跳到行控中心）
- 上架的訂單 status 設為 "pending"，等待司機接單

## 解析邏輯

車頭只需手動選「日期」「車型」「車牌」「預設金額」。**種類（接機/送機）和機場由解析器從文字自動判斷。** 同一批訂單可以混合接機和送機，也可以混合不同機場。

解析器需要從每一行文字中提取：

1. **時間**（必須有）
2. **種類**（自動判斷：接機/送機/交通接駁/包車）
3. **機場**（自動從文字中辨識）
4. **上下車點**（根據種類和機場決定 pickup/dropoff）
5. **金額覆蓋**（如果行內有金額，覆蓋預設金額）
6. **備註**（剩餘文字）

### 機場辨識表

解析器要能從文字中辨識以下機場關鍵字：

| 關鍵字 | 機場 | 代碼 |
|--------|------|------|
| 桃機、桃園機場、桃園 | 桃園國際機場 | TPE |
| 松山、松機 | 臺北松山機場 | TSA |
| 小港、高雄機場 | 高雄國際機場 | KHH |
| 清泉崗、台中機場 | 臺中清泉崗機場 | RMQ |

注意：「桃園」如果出現在「桃機到」「到桃機」「桃機-」「-桃機」的上下文中才視為機場，如果是「桃園市龜山區」這種地址就不是機場。

### 起終點連接符號

以下符號都表示起點和終點的連接，效果相同：
- 「到」：桃機**到**台中北屯
- 「-」（破折號/減號）：中壢**-**桃機
- 「→」（箭頭，偶爾出現）

### 車型自動偵測

解析器從文字中的「標題行」偵測車型。標題行只有車型關鍵字沒有時間和地點：

| 文字中的關鍵字 | 解析為 |
|--------------|--------|
| 任意 | 任意(不限車型不限車牌) |
| 任意R | 任意R牌(限租賃車) |
| 休旅 | 休旅車(7人座) |
| 大車 | 9人座 |
| 小車 | 小車(5人座) |
| 任意9座 | 9人座(不限車牌) |
| 限V | 備註保留（丟備註） |

車型標題行的作用範圍：從該行開始，到下一個車型標題行之前的所有訂單都繼承此車型。

### 套裝訂單（A1/A2/A3）

有些車頭會把一去一回綁成「一套」，標記為 A1、A2、A3 等。套裝訂單當成**一筆訂單**處理：
- 種類標記為「套裝」
- 起點和終點取第一段行程（送機那段）
- 金額為整套金額
- 備註寫完整兩段行程資訊

例如：
```
A1
04:30彰化員林到桃機
16:00桃機到台中霧峰
小車一套3300元
```
→ 一筆訂單：時間=04:30，種類=套裝，起點=彰化員林，終點=台中霧峰，金額=3300，備註=「A1套裝：04:30彰化員林→桃機 + 16:00桃機→台中霧峰 小車」

### LINE 訊息雜訊過濾

真實 LINE 群組訊息中有大量非訂單內容，解析器要先過濾掉：

**自動跳過的行（不是訂單也不是標題行）：**
- 發訊人名稱行（如「12:53 允川小魚🐟有群幫拉♡上班中ING」）— 特徵：有 emoji、有中文暱稱、沒有地點
- 純 emoji 行（如「(emoji)(emoji)(emoji)(emoji)」）
- 「已收回訊息」
- 「有群幫拉」「誠徵優質司機」等招募/廣告文字
- 分隔線（如「--------------------------------」）
- 空行

**判斷邏輯：** 一行文字如果不符合以下任何一種，就跳過：
- 有 4 位數字時間格式（如 0600、22:50）→ 可能是訂單行
- 是車型關鍵字（任意、休旅、大車、小車等）→ 車型標題行
- 有日期格式（如 4/1、4/2）→ 日期標題行
- 有「接」「送」「到」「-」等關鍵字 → 可能是訂單行
- 有「肯驛」「A1」「A2」等特殊標記 → 肯驛標題行或套裝標記
- 有金額格式（$數字 或 數字$）→ 可能包含訂單資訊

### 種類自動判斷（五種格式模式）

**模式 A：「機場 + 到/- + 地點」= 接機**
```
13:50桃機到台中北屯(安)$1800
→ 時間=13:50, 種類=接機, pickup=桃園機場, dropoff=台中北屯, 備註=(安), 金額=1800

20:50松山到台中南屯$2000
→ 時間=20:50, 種類=接機, pickup=松山機場, dropoff=台中南屯, 金額=2000

1715 桃機-文山$800
→ 時間=17:15, 種類=接機, pickup=桃園機場, dropoff=文山, 金額=800
```

**模式 B：「地點 + 到/- + 機場」= 送機**
```
06:00台中北屯到桃機$1700
→ 時間=06:00, 種類=送機, pickup=台中北屯, dropoff=桃園機場, 金額=1700

0600 中壢-桃機$500
→ 時間=06:00, 種類=送機, pickup=中壢, dropoff=桃園機場, 金額=500

0630 汐止-桃機$700
→ 時間=06:30, 種類=送機, pickup=汐止, dropoff=桃園機場, 金額=700
```

**模式 C：「接 + 地點」= 接機（機場從上下文推斷，預設桃機）**
```
0:10接土城
→ 時間=00:10, 種類=接機, pickup=桃園機場(預設), dropoff=土城

23:10 tr875 接北屯+北區 任意車2000
→ 時間=23:10, 種類=接機, pickup=桃園機場(預設), dropoff=北屯+北區, 金額=2000
```

**模式 D：「地點 + 送」= 送機（機場從上下文推斷，預設桃機）**
```
6:00板橋送
→ 時間=06:00, 種類=送機, pickup=板橋, dropoff=桃園機場(預設)

06:00 楊梅*3送桃機 反安*2 9座 1700
→ 時間=06:00, 種類=送機, pickup=楊梅, dropoff=桃園機場, 金額=1700
```

**模式 D2：「送機 + 時間 + 地點-機場」= 送機**
```
送機03:15沙鹿區-桃機
→ 時間=03:15, 種類=送機, pickup=沙鹿區, dropoff=桃園機場
```

**模式 E：交通接駁（有「上」「下」車點或「交通接駁」關鍵字）**
```
21:00交通接駁
上-高雄巨蛋
下-文心崇德捷運站
→ 時間=21:00, 種類=交通接駁, pickup=高雄巨蛋, dropoff=文心崇德捷運站
```

### 種類判斷優先順序

每行文字按以下順序判斷，命中就停止：

1. 文字中有「機場關鍵字 + 到/- + 地點」→ **接機**（模式 A）
2. 文字中有「地點 + 到/- + 機場關鍵字」→ **送機**（模式 B）
3. 文字中有「送機 + 時間 + 地點」→ **送機**（模式 D2）
4. 文字中有「接 + 地點」→ **接機**（模式 C）
5. 文字中有「地點 + 送」→ **送機**（模式 D）
6. 文字中有「送 + 機場簡稱」（如「送桃機」）→ **送機**
7. 文字中有「交通接駁」或「上」「下」車點 → **交通接駁**
8. 文字中有「包車」「時租」→ **包車**
9. 文字中有「A1」「A2」等套裝標記 + 多行行程 → **套裝**
10. 都不符合 → 標記為「解析失敗」，進入確認頁面的「需補正」區

### 解析器輸出格式

```
輸入：一行文字 + 預設值（來自 UI 選擇）
輸出：
{
  date:      預設值.日期,
  time:      從文字提取,
  type:      自動判斷（pickup/dropoff/transfer/charter/unknown）,
  pickup:    自動判斷,
  dropoff:   自動判斷,
  airport:   自動辨識的機場代碼（TPE/TSA/KHH/RMQ/null）,
  vehicle:   預設值.車型,
  plateType: 預設值.車牌,
  price:     行內金額 ?? 預設值.金額,
  notes:     剩餘文字（特殊需求如安全座椅等）,
  rawText:   原始整行文字
}
```

## UI 顯示建議（司機端訂單卡片）

司機看卡片的決策順序：種類 → 上下車點（跟現有行程有沒有順路）→ 金額 → 車型 → 時間。卡片設計要對應這個順序：

**接機卡片：**
```
┌─────────────────────────────────┐
│ 接機         3/30（日） 00:10   │  ← 種類 + 日期時間
│                                 │
│ 桃園機場 → 土城                  │  ← 上下車點（最醒目）
│                                 │
│ $700          任意R牌            │  ← 金額 + 車型
│ ─────────────────────────────── │
│ 備註：（無）                     │  ← 其他資訊
└─────────────────────────────────┘
```

**送機卡片：**
```
┌─────────────────────────────────┐
│ 送機         3/30（日） 06:00   │
│                                 │
│ 板橋 → 桃園機場                  │  ← 出發地 → 機場
│                                 │
│ $700          任意R牌            │
│ ─────────────────────────────── │
│ 備註：（無）                     │
└─────────────────────────────────┘
```

**多目的地接機卡片：**
```
┌─────────────────────────────────┐
│ 接機         3/30（日） 00:35   │
│                                 │
│ 桃園機場 → 大安+增高             │  ← 多點用 + 連接
│                                 │
│ $800          任意R牌            │
│ ─────────────────────────────── │
│ 備註：（無）                     │
└─────────────────────────────────┘
```

**交通接駁卡片：**
```
┌─────────────────────────────────┐
│ 交通接駁      3/28（六） 21:00  │
│                                 │
│ 高雄巨蛋 → 文心崇德捷運站+台中火車站│
│                                 │
│ $3,200        9人座             │
│ ─────────────────────────────── │
│ 備註：8人                       │
└─────────────────────────────────┘
```

**卡片設計原則：**
- 上下車點用箭頭「→」連接，是卡片視覺最大最醒目的一行
- 接機：上車點固定顯示機場名稱（根據平台預設或訂單指定）
- 送機：下車點固定顯示機場名稱
- 如果地點提取失敗（pickup/dropoff 為 null），顯示備註原文讓司機自行判斷

## 開發指引

1. **UI 極簡化**：車頭只需選「日期」（必選）和「預設金額」（選填）和「肯驛」（選填）。其他全部自動判斷
2. **解析器全自動**：從文字自動判斷種類、機場、車型、車牌、上下車點。支援五種格式模式 + 破折號連接符 + 車型標題行繼承 + 套裝訂單
3. **非訂單行自動過濾**：LINE 訊息中會有大量非訂單內容（暱稱、emoji、「已收回訊息」、「有群幫拉」等），解析器要能跳過這些行，只處理有時間格式的行和車型標題行
4. **確認頁面分兩區**：解析成功的可直接上架，解析失敗的（缺必備欄位）必須車頭補正後才能上架。六個必備欄位：日期、時間、種類、起點、終點、金額
5. **備註兜底**：所有非必備資訊（安全座椅、航班號、人數、行駕保、備三證、限V 等）全部丟備註
6. **保留 rawText**：永遠存一份完整原文
7. **上架即時生效**：確認上架後，訂單 status = "pending"，司機端立刻看到
8. **之後可接 AI**：初期用 regex 做模式匹配，穩定後可以加 LLM 輔助解析複雜或模糊訂單

## API 路由設計

整個貼單流程需要兩個 API：

### POST /api/orders/parse（解析，不寫入資料庫）

前端步驟二按「解析預覽」時呼叫。自動判斷種類、機場、車型。

```
Request:
{
  defaults: { date, defaultPrice, kenichiRequired },
  rawText: "任意\n2250 接板橋 700\n休旅\n0035 接三重 700\n..."
}

Response:
{
  success: true,
  data: {
    parsed: [
      { time: "22:50", type: "pickup", airport: "TPE", pickup: "桃園機場", dropoff: "板橋", vehicle: "any", price: 700, notes: "", rawLine: "2250 接板橋 700", status: "ok" },
      { time: "00:35", type: "pickup", airport: "TPE", pickup: "桃園機場", dropoff: "三重", vehicle: "suv", price: 700, notes: "", rawLine: "0035 接三重 700", status: "ok" }
    ],
    failed: [
      { rawLine: "4/1 接機CA195", status: "error", missing: ["time"], notes: "航班CA195 2人2行李 苗栗縣三義鄉廣盛村 $1600" }
    ],
    summary: { total: 3, success: 2, failed: 1 }
  }
}
```

### POST /api/orders/publish（上架，寫入資料庫）

前端步驟三按「上架到牆上」時呼叫。只寫入車頭勾選的訂單（包含成功區勾選的和補正完成的）。

```
Request:
{
  batch: { date, defaultPrice, kenichiRequired, rawText },
  orders: [
    { time: "22:50", type: "pickup", airport: "TPE", pickup: "桃園機場", dropoff: "板橋", vehicle: "any", price: 700, notes: "", rawLine: "..." },
    { time: "00:35", type: "pickup", airport: "TPE", pickup: "桃園機場", dropoff: "三重", vehicle: "suv", price: 700, notes: "", rawLine: "..." }
  ]
}

Response:
{
  success: true,
  data: {
    batchId: "clxx...",
    publishedCount: 2,
    message: "2 筆訂單已上架"
  }
}
```

注意：前端只發送車頭勾選的訂單，未勾選的不包含在 request 中。
