# 邊界案例與解析陷阱 (Edge Cases)

從真實 LINE 派單訊息中歸納的邊界情況。AI 解析時遇到下列情境，依本附錄處理。

## 陷阱 1：訊息時間戳 vs 訂單時間

派單方在 LINE 群裡貼訊息，前面常有 LINE 客戶端自動加的**訊息發送時間**：

```
15:33 小鈺 無回覆請來電 明天4/19小車
2115 CI772接文山900
```

- `15:33` 是派單方在 15:33 發送這則訊息（**不是訂單時間**）
- `2115` 才是訂單的接機時間（21:15）

**處理規則**：
- 訊息開頭的 `HH:MM 派單方暱稱` 模式 → 剝離 `HH:MM`，存到 `parse_metadata.message_sent_at`
- 訂單時間以**訂單塊內的時間**為準
- 如果整則訊息只有開頭時間，沒有訂單時間 → `rejected: missing_time`

## 陷阱 2：「送機」的隱含起終點

「送機」訊息常常**只寫起點**，因為終點預設是機場：

```
0:05三重送
```

- 起點：三重區
- 終點：桃園機場（隱含）
- 時間：00:05
- **缺**：日期、價格

**處理規則**：
- 「送」字 → `type = dropoff`，`destination = TPE`（confidence 0.85）
- 「接」字 → `type = pickup`，`origin = TPE`（confidence 0.85）
- 其他必填欄位（日期、價格）若缺 → `rejected`，rewrite_suggestion 教育格式

## 陷阱 3：群組標題的上下文繼承

派單方常用「標題 + 訂單列表」格式：

```
4/19 桃機-接
任意R-小車/大車皆可
20:10 接-彰化社頭 $1800
00:55 接-台中西屯 $1600
```

- `4/19` 是整批訂單的日期
- `桃機` 是整批訂單的機場
- `任意R` 是整批訂單的車型
- 個別訂單只寫時間、地點、價格

**處理規則**：
- **嚴格模式**：標題上下文不可繼承，每筆訂單必須自己寫日期 → 拆出來的訂單 `rejected`，rewrite_suggestion 提供合併版本
- **寬鬆模式**：可繼承標題的日期/機場/車型，但 confidence 降至 0.6，標 `needs_review`
- 預設用嚴格模式，明確要求派單方在每筆訂單明寫日期

> **重要**：因為這是派單方最常見的習慣，AI 在 rewrite_suggestion 中應該**提供整批改寫範例**，而不是逐筆退回。例如：
> ```
> ⚠️ 標題日期無法繼承到個別訂單
> ✏️ 建議改寫整批：
> `4/19 20:10 接機 彰化社頭 1800`
> `4/20 00:55 接機 台中西屯 1600`
> 📖 為什麼：跨日訂單放在同一段時，標題日期常被誤判
> ```

## 陷阱 4：跨日「後天」「明天」的歧義

```
後天4/18 大車TVG $1900
15:10大安送+17:05接中正+中正
```

- 「後天 4/18」假設訊息發送日為 4/16
- 但如果派單方已經貼了 5 次（從 4/16 貼到 4/18），「後天」就會錯
- 而且 `4/18` 已經是明確日期，「後天」是冗餘描述

**處理規則**：
- 如果同時有相對詞（明天/後天）和明確日期（4/18）→ 採用**明確日期**，confidence 0.95
- 如果只有相對詞 → `rejected`，rewrite_suggestion 要求明寫日期
- 不論哪種情況，AI 都不應該用「訊息發送時間」+ 相對詞推算日期（風險太高）

## 陷阱 5：套裝訂單「一套不拆」

```
=1-2==一套不拆
03:30 台中東區-送 (進口車) $1800
18:45 接-台中西屯 (進口車) $1800
```

派單方意圖：**同一台車先送機、再接機**，希望司機接下整套。

**處理規則**：
- **拆成兩筆獨立訂單**（不合併）
- 兩筆訂單都標 `bundle_intent: true` 與 `bundle_ref: "=1-2="`（保留派單方原始套裝編號，前端可顯示視覺關聯）
- 在 `rewrite_suggestion` 教育：
  ```
  ⚠️「一套不拆」是 LINE 時代的權宜寫法
  ✏️ 建議改寫：在 gogmo App 上分別建立這兩筆訂單，再點「綁定派單」按鈕
  📖 為什麼：平台的綁定派單會強制司機接下整套，比文字說明可靠
  ```
- 注意：不主動合併金額（不要把 `$1800 + $1800` 變 `$3600`）

## 陷阱 6：多停點訂單

```
A80
23:20接機JX805 
1.南港區玉成街
2.汐止區福德一路
休旅💲1000
```

一趟車接機後，先到南港，再到汐止。

**處理規則**：
- `destination` 改為陣列：`["南港區玉成街", "汐止區福德一路"]`
- 或標記 `multi_stop: true` 與 `stops: [...]`
- confidence 維持 0.9（這個格式很明確）
- 不需要 rewrite_suggestion（派單方寫法已經夠清楚）

## 陷阱 7：條件式加價

```
$850（小車）
（如超過23:00搭車+$100深夜自取）
```

**處理規則**：
- `price.value = 850`（基本價）
- `special_requirements` 增加標籤 `late_night_surcharge:100:23:00`
- 不主動加總（不要把 850 變 950），讓 App 在派單時依實際抵達時間動態計算

## 陷阱 8：全域預設價

```
任意R $700
0:05三重送
0:30中和送
3:50文山送
5:00汐止國興街送 $800
```

派單方意圖：開頭 `$700` 是這批的預設價，個別訂單除非另寫，都用 700。

**處理規則（這是嚴格模式的核心測試）**：
- **不繼承全域預設價**
- 個別訂單缺價格 → `rejected`，rewrite_suggestion：
  ```
  ⚠️ 缺日期、缺價格
  ✏️ 建議改寫：`4/19 00:05 送機 三重區 700`
  📖 為什麼：每筆訂單的價格獨立，不要靠開頭的「預設價」推測，避免錯派
  ```
- 即使派單方抗議，平台立場應該是：**請在 App 設定「派單預設價 = 700」**，不要寫在訊息裡

## 陷阱 9：重複貼單

派單方擔心訊息被洗掉，會把同一批訂單在 30 分鐘內貼 3-7 次。

**處理規則**：
- **不去重，全部解析**
- 在 `parse_metadata.duplicate_warning` 標記偵測到的重複數量：
  ```json
  "duplicate_warning": {
    "detected": true,
    "count": 5,
    "first_seen_at": "2026-04-18T15:42:00",
    "last_seen_at": "2026-04-18T18:01:00",
    "similarity_score": 0.98
  }
  ```
- 由前端 / 業務邏輯決定：
  - 顯示「⚠️ 偵測到重複貼單，是否合併？」按鈕
  - 或自動隱藏舊版本，只顯示最新
  - 或對該派單方計入「重複貼單次數」，觸發收費

## 陷阱 10：雜訊與系統訊息

直接過濾不解析：

| 類型 | 範例 |
|---|---|
| 退出群組 | `𝓨𝓪𝓸 𝔂𝓪𝓸🦋 已退出群組` |
| 純廣告 | `✈️包車找我❤️`、`好友已🈵` |
| 群組規則 | `備三證、接單@後私訊` |
| 結尾話術 | `可以的請私謝謝`、`可以的@我🙏🏻` |
| 純表情 | 整則只有 emoji |

存到 `parse_metadata.noise_filtered_count`，不產生 order 也不產生 rejection。

## 陷阱 11：純接駁（非機場）

```
4/19。00:30 接駁趟
客四
松山區-高雄
🈯️大車 5700$
```

不是接機也不是送機，是城際接駁。

**處理規則**：
- `type = transfer`（非 pickup / dropoff）
- `origin = 松山區`、`destination = 高雄`
- `airport = null`
- 其他欄位照常

## 陷阱 12：「客X+件Y」的人數行李格式

```
大車5人7件💲1000
九座8人5件💲1150
```

- `5人7件` = 5 個乘客，7 件行李
- 用於司機評估能否載完

**處理規則**：
- 解析為 `passenger_count: 5`、`luggage_count: 7`
- 存到 `special_requirements` 或獨立欄位
- 是輔助資訊，不影響核心 6 欄位 confidence

## 處理優先級

當一筆訂單同時觸發多個陷阱時，依序處理：

1. **過濾雜訊**（陷阱 10）
2. **剝離訊息時間戳**（陷阱 1）
3. **判斷訂單塊邊界**（空行 / 編號 / 分隔線）
4. **檢查必填欄位**（日期、時間、類型、地點、價格）
5. **應用隱含規則**（送機/接機 → 機場、多停點 → 陣列）
6. **檢測套裝/多停點/重複**（陷阱 5、6、9）
7. **計算 confidence 與 parse_status**
8. **生成 rewrite_suggestion**（若 status != accepted）
